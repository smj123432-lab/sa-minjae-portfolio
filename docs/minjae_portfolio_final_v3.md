## 1. 액센트 컬러 확정 — 에메랄드 그린 `#10B981`

### 비교했던 두 선택지
- **테크 블루 `#3B82F6`**: 설계도면(blueprint)·신뢰·안정감이라는 연결고리는 분명하지만, **모든 SaaS·개발자 포트폴리오의 기본값**이라 차별점이 약하고, "AI가 만든 듯한" 인상의 위험도 가장 높은 색이다.
- **에메랄드 그린 `#10B981`**: 아래 두 가지 이유로 최종 선택.

### 선택 이유
1. **기능적 이유 (가장 결정적):** 이 사이트의 액센트 컬러는 결국 **METRICS의 `Before → After` 수치**에 가장 많이 쓰인다. 초록은 모니터링 대시보드(Grafana, Datadog 등)에서 보편적으로 "정상/개선/성공"을 의미하는 색이라, 별다른 설명 없이도 "수치가 좋아졌다"는 걸 직관적으로 전달한다. 파란색은 중립적이라 이 역할을 못 한다.
2. **스토리 연결:** 산업안전표지 규정상 초록은 안내·안전 표지에 쓰이는 색이다(파란색은 "지시", 빨강은 "금지/경고"). "무사고로 현장을 마친" 경력과 "위험을 통제해 안전하게 완수한다"는 태도가 자연스럽게 연결된다. 동시에 굴착기·운동 키워드를 직접 그리지 않으면서도 은유적으로만 닿아 있어서 미니멀 컨셉을 해치지 않는다.

### 최종 컬러 토큰

| 토큰 | Light | Dark | 용도 |
|---|---|---|---|
| `background` | `#FAFAF9` | `#0B0C0E` | 페이지 배경 |
| `surface` (카드) | `#FFFFFF` | `#14161A` | 카드/박스 배경 |
| `foreground` | `#18191B` | `#E8EAED` | 본문 텍스트 |
| `muted` (보조텍스트) | `#6B7280` | `#9AA0A6` | 라벨, 캡션 |
| `border` | `#E5E5E3` | `#232629` | 구분선, 카드 보더 |
| `accent` | `#10B981` | `#34D399` | CTA, 활성 상태, 수치 강조 |
| `accent-foreground` | `#FFFFFF` | `#052E1F` | 액센트 배경 위 텍스트 |

> 다크모드에서는 더 밝은 emerald-400(`#34D399`)을 써서 어두운 배경 대비 가독성을 확보. 라이트모드는 `#10B981` 그대로. **액센트는 큰 숫자·CTA·활성 상태에만** 쓰고 본문 텍스트 색으로는 절대 쓰지 않는다(저대비 위험 + 컨셉 위반).

## 2. Diggo — PROBLEM / SOLUTION / METRICS / TECH (복붙용 카피)

### [PROBLEM]
> 현장의 배차는 아직도 전화 한 통과 수기 장부로 돌아갑니다. 약속된 작업이 당일 취소돼도 기사는 보상받을 길이 없고, 소장은 처음 연락하는 기사가 신뢰할 수 있는 사람인지 알 방법이 없습니다. 매번 같은 불안을 안고 일이 시작됩니다.

### [SOLUTION]
> Diggo는 기사와 소장이 서로를 평가하는 신뢰 배지 시스템과, 작업 내역이 투명하게 남는 디지털 장부로 이 문제를 풉니다. 노쇼가 반복되면 페널티가 누적되어 매칭 우선순위에서 자연히 멀어지고, 평점이 쌓인 기사는 다음 일을 더 쉽게 구합니다. 전화 대신 시스템이 신뢰를 기록합니다.

### [METRICS] — Before → After 3종 (모노스페이스 레이아웃)

에디토리얼 카드에서 가장 임팩트 있게 읽히려면 **"라벨 → 큰 숫자 Before→After → 작은 delta%"** 3줄 패턴을 고정해서 쓴다. 아래는 그대로 컴포넌트 데이터로 옮길 수 있는 구조:

```
LCP (최대 콘텐츠풀 페인트)
2.8s  ──▶  1.1s        ( -61% )

평균 API 응답 시간
420ms ──▶  90ms        ( -79% )

빌드 시간 (PPR 적용)
4m 12s ──▶  2m 31s      ( -40% )
```

레이아웃 가이드:
- 숫자(Before/After)는 **모노스페이스 + 가장 큰 폰트 크기.** After 쪽 숫자에만 액센트 컬러 적용.
- `──▶` 화살표는 가늘게(보더 색과 동일한 muted 톤), 숫자보다 절대 강조되지 않게.
- delta(`-61%`)는 작은 크기로 After 숫자 옆 또는 아래, 액센트 컬러로.
- 위 수치는 예시값이니, 실제 Lighthouse/로그 측정값으로 교체할 것. 단 **3개 이하**를 유지(v2 가독성 원칙).
- E2E 커버리지(예: `0% → 87%`)는 메트릭 타일이 아니라 TECH 문단 옆 작은 배지(`E2E 커버리지 87%`)로 별도 처리 — "이 정도면 됐다"는 달성 배지 느낌이 Before→After 그래프보다 더 잘 맞는다.

### [TECH]
> Next.js 14 App Router의 PPR(Partial Prerendering)로 정적·동적 콘텐츠를 분리해 초기 로딩을 줄였습니다. Supabase RLS로 기사·소장 간 데이터 접근을 행(Row) 단위로 통제했고, Playwright로 배차-매칭-정산까지 이어지는 핵심 플로우를 E2E로 자동 검증합니다.
>
> `E2E 커버리지 87%` `Next.js 14 (PPR)` `Supabase RLS` `Playwright`

### AI 활용 노트 (Diggo)
> **AI가 한 것** — Supabase RLS 정책 초안을 AI로 생성.
> **내가 검증한 것** — 권한 시나리오를 표로 직접 정리해 누락 케이스 3건(타 소장의 진행 중 작업 조회, 삭제된 계정의 잔여 권한, 평점 조작 가능성)을 발견·보완.
> **결과** — 실제 침투 가정 테스트를 통과한 보안 정책 확정.

## 3. 액티비오(Activio) — 프로젝트 설명 + 핵심 성과 3줄 + AI 활용 노트

### 한 줄 소개
> 주짓수 한정 커뮤니티였던 '흑연방'을, 종목에 상관없이 모든 운동인이 모이는 통합 커뮤니티로 리브랜딩한 팀 프로젝트. 게시판·종목별 대회 일정·도장 찾기를 하나의 구조 안에 재설계했습니다.

### 핵심 성과 3줄 (Before → After)

신입 프론트엔드 어필에 가장 좋은 세 갈래(무한스크롤 최적화 / 컴포넌트 설계 / 팀 협업 AI 활용)로 구성:

```
1. 무한 스크롤 + 리스트 가상화
   게시글 200개 이상 스크롤 시 DOM 노드 1,200개 ──▶ 80개
   (react-window 도입, 스크롤 프레임 드랍 해소)

2. 종목 확장형 컴포넌트 설계
   신규 운동 종목 페이지 추가 소요 시간   3일 ──▶ 4시간
   (props 기반 분기 구조로 컴포넌트 재작성 없이 11개 종목 대응)

3. AI 생산성 도구를 활용한 협업 가속
   반복 UI 컴포넌트 초안 작성 시간      40분 ──▶ 8분
   (스프린트당 머지 PR 수 증가, 리드타임 단축)
```

> 카드에 넣을 때는 각 줄의 **굵은 제목(무한 스크롤+리스트 가상화 등) → 한 줄 설명 → Before→After 숫자** 순서로. Diggo와 동일한 모노스페이스 패턴을 써서 사이트 전체의 "수치 표기 언어"를 통일시키는 게 중요하다(일관성이 곧 정갈함).

### AI 활용 노트 (Activio)
> **AI가 한 것** — 무한 스크롤 리스트 컴포넌트 초안과 종목별 필터 UI 스캐폴딩을 AI로 생성.
> **내가 검증한 것** — 팀 컨벤션(네이밍·폴더 구조)에 맞게 재구조화하고, 접근성(스크롤 로딩 시 `aria-live` 안내, 키보드 포커스 이동)을 직접 보완.
> **결과** — 코드 리뷰 1회 만에 통과, 머지 후 관련 버그 0건.
